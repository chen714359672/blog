vue多页面项目的构建以及文件的划分

vuePress-theme-reco 小铭同学    2020 - 2022

vue多页面项目的构建以及文件的划分

小铭同学 2021-04-10 vue

介绍

基于vue的多项目多页面拆分以及构建

前言

最近公司准备新开展一个b2c电商项目，考虑到电商项目经常有活动更新，或者组件更新，会频繁修改，同时可能进行每个页面的灰度发布，单页面的话每次都得全部重新打包，因此把项目拆分为多页面。每个功能模块属于一个html页面来渲染，去掉router路由控制，路由由后端全权控制。

经过2天时间的目录层次划分，以及打包文件的配置，终于实现了多项目下多页面，从模块上可以每个模块单独打包，从模块功能上可以每个功能单独打包，实现了点到面的聚合。npm运行指令以及打包指令上，也可以一行命令分别打包、运行不同项目-不同模块。

# 需求分析

# 目录结构划分

在大的目录层次上，多项目跟单项目并没有多大的差别，同样是常见的vue.config.js ，package.json，src目录

src目录开始，划分出shop目录，user目录，common目录。

一个是卖家商家端项目，一个是买家用户端项目，以及一个公用的common目录，里面包含了一些通用的组件库，样式，以及js文件。

进入shop卖家项目来看，跟普通单项目不同的是，在这层目录下，去掉了router目录，App.vue文件，main.js文件。项目下是按模块来进行划分，因为每个模块单独打包渲染html，控制模块之间的跳转，直接用a标签跳转到新的html页面就可以了。

# 路由router的控制

可以看到的是，原来在项目目录下的App.vue文件，main.js文件转移到了每个模块下的目录上，因为只有这样，才能每个模块都能打包生成自己的html页面。

vuerouter文件，也不需要其他配置了，引入只是为了设置history模式，以及能获取路由参数那些，当然，你也可以直接不引用vue-router，通过js来获取url参数。

在这之间，有一点特别需要注意的是，每个目录下的main.js，最好不用引入全局组件。把所有能按需引入的组件，都在各自的vue文件下，按需引入。因为考虑到功能单独打包，但又可能是模块打包，如果功能下的main.js引入了，那么到时候模块的main.js也需要引入，不然模块的main.js打包就会报错了。如下图：

project接收的是项目名，modulePath接收的是模块或者功能名

因为每个模块都有main.js，同时要想运行或者打包时不用每次都手动引入，就需要写个函数来实现自动引入，其中需要用到关键的glob来匹配文件，需要自行npm一下。

新建一个js文件，来导出3个方法，让vue.config.js去接收配置项pages，outputDir，publicPath

例如：

const path = require('path')
const glob = require('glob')

const name = process.argv[process.argv.length - 1].replace('--', '').split('-')[0]
const project = process.argv[process.argv.length - 1].replace('--', '').split('-')[1]

// pages
const generateEntries = () => {
  // 默认查询多页面地址
  const PATH_ENTRY = path.resolve(__dirname, `./src/${name}/modules/**`)
  // 约定构建出的页面用folder名字，默认入口为每个页面的main.js
  const entryFilePaths = glob.sync(PATH_ENTRY + '/**/main.js')
  const entry = {}

  entryFilePaths.forEach((filePath) => {
    const FILENAME = filePath.match(/([^/]+)\/main\.js$/)[1]
    entry[FILENAME] = {
      entry: filePath,
      template: 'public/index.html',
      filename: 'index.html',
      // title可不传，每个页面单独设置
      title: FILENAME
    }
  })

  return entry
}

module.exports = {
  generateEntries
}
